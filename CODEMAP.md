## 5. Execution Flows

### Startup Flow

1. User invokes `codetalk <command> [args]` in terminal.
2. `src/index.ts` (built by `yargs`) parses command and arguments.
3. If command requires config (`scan`, `plan`, `exec`, `ask`), load `~/.codetalker/config.json` via `utils.readConfig`.
   - If missing or invalid, error with message to run `codetalk config`.
4. If command requires `CODEMAP.md` (`plan`, `exec`, `ask`, `check`), verify map existence and freshness via `utils.hasGit` and `utils.getChangedFiles`. If stale or missing → exit 1.
5. Dispatch to the corresponding handler in `src/handlers.ts`.

### Command Flow

| Command | Handler | Key Steps |
|---------|---------|-----------|
| `help` | `constants.printHelp` | Print built‑in help text |
| `init` | `initMap` | Write template `CODEMAP.md` via `utils.buildTemplate` |
| `config` | `configure` | Interactive prompt using `readline`; read/write config |
| `scan` | `scanRepo` | 1. `utils.collectSourceFiles` / `indexer.buildSymbolIndex`<br>2. `buildInspectionPlan`<br>3. `splitFilesForAgents`<br>4. `runReviewerAgents` (parallel LLM calls via `api.callChatCompletion`)<br>5. `mergerPrompt` / aggregation<br>6. Output result |
| `map` | `writeMap` | 1. `runArchitectureScan` (uses indexer + tools)<br>2. `utils.buildMap`<br>3. Write `CODEMAP.md` |
| `plan` | `planChange` | 1. `readMapForContext`<br>2. Build prompt via `prompts.planSystemPrompt`<br>3. `api.callChatCompletion` or `streamChatCompletion`<br>4. Parse plan (markdown)<br>5. Write to `CODEPLAN.md` via `writePlan` |
| `exec` | `execution` | 1. Read `CODEPLAN.md`<br>2. Back up current source files to `~/.codetalker/backups/<timestamp>/`<br>3. `parseExecChangeSpecs` to extract steps<br>4. For each step, call LLM (parallel with `runLimited`)<br>5. Apply edits via `fs.writeFile`<br>6. `runSemanticSync` to update `CODEMAP.md` |
| `ask` | `askCodebase` | 1. Read map + optional scan context<br>2. `prompts.askSystemPrompt` + user question<br>3. `api.streamChatCompletion` (or non‑streaming)<br>4. Output answer to stdout |
| `check` | `checkMap` | 1. `utils.hasGit`<br>2. `utils.getChangedFiles`<br>3. Compare timestamps; exit 0 if fresh, 1 if stale |
| `rollback` | `listBackups`/`rollbackTo` | 1. With `--list`: list backups in config dir<br>2. With backup ID: copy files from backup to project root |

### Plan Flow (from user request to CODEPLAN.md)

1. User runs `codetalk plan "Implement feature X"`.
2. `planChange` reads `CODEMAP.md` via `readMapForContext`.
3. Constructs prompt: system from `planSystemPrompt` + user request + map context.
4. Sends to LLM via `api.runPrompt` or streaming.
5. LLM returns markdown plan with file paths, actions, and descriptions.
6. `writePlan` writes the plan to `CODEPLAN.md` (default output).
7. If streaming, progress is shown to terminal.

### Exec Flow (from plan to file edits)

1. User runs `codetalk exec`.
2. `execution` reads `CODEPLAN.md`.
3. Creates backup: copies all files mentioned in plan to `~/.codetalker/backups/<timestamp>/`.
4. Parses plan into `ChangeSpec[]` (file, action, content description).
5. For each spec, LLM is called (with `createExecEditorPrompt`) to generate actual new file content.
6. Edits are applied concurrently (up to `parallel` limit) via `fs.writeFile`.
7. After all edits, `runSemanticSync` calls LLM to update `CODEMAP.md` with any changed semantics.
8. Final `CODEMAP.md` is written.

### Scan Flow (from repo to CODEMAP.md)

1. User runs `codetalk scan` or `codetalk map`.
2. `collectSourceFiles` walks repository tree, filtering by extensions.
3. `buildSymbolIndex` extracts exports and symbols (via grep?).
4. `buildInspectionPlan` assigns priorities to files.
5. `splitFilesForAgents` divides files among reviewer agents.
6. For each group, LLM is called with appropriate `reviewerPrompt` (Low/Medium/High/Full).
7. `mergerPrompt` combines results into a coherent `ScanReport`.
8. `writeMap` (if `map` command) or `formatScan` (if `scan` command) generates output.

### Error Flow

- **Missing config**: `configure` handler prints instructions and exits 1.
- **Stale/missing `CODEMAP.md`**: `checkMap` exits 1 with explanatory message.
- **LLM API failure**: `callChatCompletion` retries up to 3 times (configured in constants). On final failure, error is propagated to handler, which prints error and exits 1.
- **File write failure**: Backup already taken; user can `rollback`. Handler prints error and exits 1.
- **Invalid plan syntax**: `parseExecChangeSpecs` throws; `execution` catches, prints error, exits 1. No changes applied.
- **Git not found**: `hasGit` returns false; `checkMap` and `getChangedFiles` will report error or fall back to timestamp comparison.

### Rollback Flow

1. User runs `codetalk rollback --list` to see available backups.
2. Or `codetalk rollback <backup-id>`.
3. `rollbackTo` reads backup directory, copies each backed‑up file back to its original location.
4. Overwrites current files (no confirmation). Restored files are not auto‑synced to map.

### Caching

- **Symbol index** (`buildSymbolIndex` / `saveIndex` / `loadIndex`): An optional JSON cache of symbols per file. Used by `indexer` to avoid re‑scanning unchanged files. Stored at `~/.codetalker/index.json`. Lifetime: until next `scan` or `map`. Invalidated when source files change (timestamp check).
- **LLM responses**: Not cached by default. Each invocation re‑contacts the API.
- **Backup snapshots**: Kept indefinitely until manual cleanup.
