## Architecture

`codetalk-cli` is a command-line tool (package name `codetalk-cli`, version 0.1.4) for analyzing codebases, producing living semantic maps, and executing safe, planned changes via LLM agents. It is written in TypeScript and compiled to `dist/index.js` (two binaries: `codetalk` and `codetalk-cli`). Additionally, a VS Code extension is provided in `src/vscode/` for dashboard UI and command integration.

**Main execution path**: The user invokes a CLI command (`help`, `init`, `config`, `scan`, `map`, `ask`, `plan`, `exec`, `check`, `rollback`, `version`, `semantic`). The entry point (`src/index.ts`) parses arguments and dispatches to a handler in `src/handlers.ts`. Each handler orchestrates the required logic: config loading via `src/utils.ts`, LLM API calls via `src/llm/index.ts` (LlmPortal), tool execution via `src/tools/index.ts`, symbol indexing via `src/indexer.ts`, LSP integration via `src/lsp/`, semantic analysis via `src/semantic.ts`, progress display via `src/panel.ts`, interactive config TUI via `src/tui/config.ts`, and core command orchestration via `src/core/commands.ts`. The file walkers honor `.gitignore` so scans, indexes, and code search stay aligned with Git's ignore rules.

**VS Code extension**: `src/vscode/extension.ts` activates and registers a webview view provider (`CodetalkViewProvider`) that renders a dashboard (`src/vscode/webview.ts`) and communicates messages back to the CLI via the `CodetalkRunner` embedded in the extension context. The shim file `src/vscode-shim.d.ts` provides ambient type declarations for the `vscode` module to avoid a real dependency during development.

**Agent skill definitions**: `agents/openai.yaml` defines skill metadata (display name, default prompt) for the OpenAI‑compatible provider.

**CI**: `.github/workflows/ci.yml` is present (noted in scan output) but not loaded by the CLI; it is used by CI runners only.

**Major components**:
- **CLI dispatcher** (`src/index.ts` + `src/constants.ts`): argument parsing, help generation, command routing.
- **Core command runner** (`src/core/commands.ts`): `runCodetalkCommand`, `dispatchCommand`, `captureProcessOutput`, `getDashboardState` – used by both CLI and VS Code extension.
- **Handlers** (`src/handlers.ts`): implementation of all commands, plus sub‑orchestrators for scanning, planning, execution, semantic analysis, and semantic map synchronization.
- **LLM Portal** (`src/llm/index.ts`, `src/llm/base-handler.ts`, `src/llm/openai-handler.ts`, `src/llm/anthropic-handler.ts`, `src/llm/registry.ts`, `src/llm/types.ts`): provider‑agnostic LLM client; supports non‑streaming and streaming chat completions, tool calls, token tracking, colored stream output, error types (LlmHttpError, LlmProtocolError, LlmTimeoutError, LlmTransportError).
- **Legacy API client** (`src/api.ts`): still present but superseded by `LlmPortal`; provides `callChatCompletion`, `streamChatCompletion`, `callWithTools` wrappers.
- **Tool system** (`src/tools/`): read, grep, ls, glob, stat, git_log, search – invoked by the agent during planning/execution.
- **AST extractors** (`src/ast/`): extract symbols from TypeScript, Python, C++, and assembly files.
- **Indexer** (`src/indexer.ts`): builds in‑memory symbol indices from source trees, used by `search` and `grep` tools. Now also uses LSP pool.
- **LSP client** (`src/lsp/`): JSON‑RPC LSP client per language server; `LspPool` coordinates extraction across files/extensions.
- **Semantic analyzer** (`src/semantic.ts`): extracts detailed function/method semantics (purpose, inputs, outputs, side effects, failure modes) for the `semantic` command. Now includes parallel task runners (`runSemanticTasks`, `resolveSemanticParallel`) and progress formatting utilities.
- **Prompt templates** (`src/prompts.ts`): system prompts, tool definitions, and prompts for ask, plan, semantic extraction, reviewer agents, etc.
- **Config** (`src/utils.ts`): read/write JSON config file (`~/.codetalker/config.json`), manage API URL, key, model; also `src/tui/config.ts` for interactive TUI configuration.
- **Semantic map generator** (`src/utils.ts`): produces `CODEMAP.md` from repository structure (`buildMap`, `buildTemplate`, `buildChangeSync`, `buildRepositoryEvidence`), scans config, package info, CI, semantic maps, module roles.
- **VS Code extension** (`src/vscode/extension.ts`, `src/vscode/viewProvider.ts`, `src/vscode/webview.ts`): dashboard UI for viewing map status, running commands, and receiving events.
- **Build scripts** (`scripts/build.mjs`, `scripts/test-cli.mjs`, `scripts/test-rollback.mjs`): build and test automation.
- **Reference documents** (`references/repo-semantic-map.md`, `references/semantic-map-template.md`): canonical semantic map and reusable template.
- **Shim declarations**: `src/vscode-shim.d.ts` provides ambient types for the `vscode` module so that the extension can be compiled without the real VS Code types package.

**Scale**: 60 source files (43 TypeScript, 14 Markdown/other, 3 JavaScript) plus supporting YAML and config files. Primary language: TypeScript. Designed for single-user CLI use, no server.

**Design philosophy**: Agent‑centric; the semantic map (`CODEMAP.md`) is the behavioral contract for LLM agents. Change safety via explicit protocol (plan → execute → sync). Rollback support via backups. Semantic analysis augments the map with function‑level metadata.

## API Surface

### CLI Commands

| Command | Purpose | Flags |
|---------|---------|-------|
| `codetalk help` | Show commands and user workflow. | (none) |
| `codetalk init` | Create a semantic map template (`CODEMAP.md`). | (none) |
| `codetalk config` | Interactive configure API URL, API key, and model. | (none) |
| `codetalk scan` | Analyze repository and produce a living semantic map. | `--json`, `--stream`, `--timeout MS`, `--depth low\|medium\|high\|full`, `--out PATH`, `--parallel N`, `--write` |
| `codetalk semantic` | Extract detailed function and method semantics into CODEMAP.md. | `--parallel N\|MAX`, `--timeout MS`, `--out PATH` |
| `codetalk map` | Generate a baseline semantic map from repository structure. | `--out PATH` |
| `codetalk ask "message"` | Answer codebase questions from map and scan context. | `--stream` |
| `codetalk plan "request"` | Generate a safe implementation plan and write to disk. | `--stream`, `--out CODEPLAN.md` |
| `codetalk exec` | Execute a CODEPLAN.md: apply all file changes in parallel via LLM (auto‑syncs map). | `--plan CODEPLAN.md`, `--parallel 4`, `--stream`, `--skip-gate`, `--diff` |
| `codetalk check` | Fail when the semantic map is missing or stale. | (none) |
| `codetalk rollback` | List backups or restore files from a backup. | `--list`, `<backup-id>` |
| `codetalk version` | Print the CLI version. | (none) |

### VS Code REST‑like API (via Webview Messages)

The VS Code extension communicates using JSON messages over the webview `postMessage` / `onDidReceiveMessage` channel. The dashboard sends commands (string) and receives events (`CodetalkEvent`) with status, stdout, stderr, and progress. See `src/core/commands.ts` for the `CodetalkCommandRequest` and `CodetalkCommandResult` types.

### Exported Package API

No public npm package API; all functionality is CLI‑driven or VS Code extension‑driven.

## Execution Flows

### Startup Flow
1. User runs `codetalk <command>` (or `codetalk-cli`).
2. `dist/index.js` invokes `src/index.ts::main()`.
3. `main()` calls `utils.parseOptions(process.argv)` to extract command and flags.
4. Matches command against `constants.COMMANDS` list.
5. If no match, prints help and exits.
6. Calls the corresponding handler function from `src/handlers.ts`.
7. Handler optionally loads config via `utils.readConfig()`.
8. Handler proceeds with its specific flow.

### VS Code Extension Flow
1. VS Code activates the extension (`activate()` in `src/vscode/extension.ts`).
2. Registers the `CodetalkViewProvider` as a webview view provider.
3. The webview dashboard appears; it sends a `getDashboardState` event to load initial state.
4. `CodetalkViewProvider.postState` returns `DashboardState` (config, map status, git status, backups).
5. User issues commands via dashboard buttons; messages are sent to `handleMessage`.
6. `handleMessage` calls `runCodetalkCommand` which captures stdout/stderr and emits events back to the webview.
7. Progress updates are forwarded via `onProgress` callback of `MissionPanel`.

### Command Flow
- **init**: `initMap()` → `utils.buildTemplate()` → writes to `CODEMAP.md`.
- **config**: `configure()` → `tui/config.configureTui()` → interactive menu → writes config.
- **scan**: `scanRepo()` → `utils.collectSourceFiles()` → `utils.buildScanReport()` → optionally calls `runArchitectureScan()` which uses LSP pool and LLM reviewer agents → `utils.buildMap()` → writes `CODEMAP.md`.
- **semantic**: `semanticMap()` → reads current map → calls `semantic.runSemanticMap()` which builds inventory, runs parallel workers (`runSemanticTasks`) to analyze each function via LLM, merges cache, renders semantic section, and writes updated map.
- **ask**: `askCodebase()` → reads map + builds evidence → sends to LLM (stream or non‑stream) → displays answer.
- **plan**: `planChange()` → reads map + builds evidence → sends to LLM with tools → writes `CODEPLAN.md`.
- **exec**: `execution()` → reads `CODEPLAN.md` → parses specs → creates backup → spawns parallel editor agents (via LLM) → validates with gatekeeper → applies changes → auto‑syncs map.
- **check**: `checkMap()` → reads map and checks if new source files are older than map; exits non‑zero if stale.
- **rollback**: `listBackups()` or `rollbackTo(id)` → restores backup directory content.
- **version**: `printVersion()` → outputs constant.

### Semantic Map Sync Flow
1. Triggered after `exec` (auto) or separately via `syncMap()`.
2. Gets list of changed files via `utils.getChangedFiles()`.
3. Sends `semanticSyncPrompt` to LLM with old map and changed files.
4. LLM returns updated map text.
5. `replaceSection()` updates `CODEMAP.md` in place.

### Semantic Analysis Flow (updated)
1. `semanticMap()` in handlers calls `runSemanticMap(options)`.
2. Collects source files via `collectSourceFiles`.
3. Builds baseline map if none exists (`buildMap`).
4. Builds symbol index via `buildSymbolIndex` and saves it.
5. Extracts hierarchical symbols via `LspPool.extractAll`.
6. Builds function inventory via `buildSemanticInventory` (uses LSP or heuristic excerpts).
7. Loads semantic cache from `.codetalk/semantic/manifest.json`.
8. Runs parallel workers via `runSemanticTasks` with concurrency controlled by `resolveSemanticParallel`.
9. Each worker calls `analyzeFunction`:
   - Checks cache hit by fingerprint; if unchanged, skips LLM call.
   - Otherwise, calls `callChatCompletionMessages` with extraction prompt.
   - Parses response via `parseSemanticJson` and normalizes into `SemanticFunctionRecord`.
10. Merges results into cache via `mergeSemanticCache`.
11. Renders `## Functions` section via `renderSemanticSection`.
12. Updates `CODEMAP.md` via `replaceSection` and `refreshGeneratedHeader`.
13. Saves updated cache and writes map.

### Error Handling
- Missing config: commands that need LLM exit with error message (or prompt for config).
- LLM API failures: caught by handler, retry logic in `callWithTools` and `callChatCompletionMessages` (limited retries).
- Tool execution failure: returns `ToolResult` with `success: false`.
- Plan execution failure: editor agents retry up to `MAX_RETRIES` (3).
- Gatekeeper rejection: loops back to coordinator for refinement, up to `MAX_GATE_RETRIES` (3).
- LSP server failure: `LspPool` catches start errors, marks server as failed, returns empty symbols for those files.
- Timeouts: all LLM calls have configurable timeout (env `CODETALKER_TIMEOUT_MS` or default 120s); tool calls may also have timeouts.
- VS Code extension errors are shown via `vscode.window.showErrorMessage`.

## Data Flow

### Data Sources
- **File system**: source files, config file (`~/.codetalker/config.json`), git repository.
- **LLM API**: OpenAI‑compatible or Anthropic API (configured via `apiUrl`).
- **User input**: CLI arguments, interactive prompts (config TUI, plan request text), VS Code webview messages.
- **VS Code**: workspace folders, extension context, configuration.

### Data Transformations
1. **File → Symbol Index**: `collectSourceFiles()` → `buildSymbolIndex()` (AST + LSP) → `SymbolIndex` (mapping file → exports, functions, imports, types).
2. **Symbol Index → Evidence**: `buildRepositoryEvidence()` truncates and formats for agent context.
3. **File + Map + LLM → Semantic Map**: `runSemanticMap()` → inventory items → parallel LLM workers generate `SemanticFunctionRecord` per function → `renderSemanticSection()` → updated map.
4. **Plan Request → CODEPLAN.md**: `planChange()` → LLM with tools → YAML plan file.
5. **CODEPLAN.md → File Edits**: `execution()` → parse YAML → edit via LLM editor agents → apply diffs.
6. **Map + Diff → Updated Map**: `syncMap()` → LLM updates map sections.
7. **VS Code Command Request → Result**: `runCodetalkCommand` captures stdout/stderr and returns structured result to webview.

### Data Sinks
- `CODEMAP.md` – written by `scan`, `semantic`, `map`, `syncMap`.
- `CODEPLAN.md` – written by `plan`.
- `~/.codetalker/config.json` – written by `config`, `readConfig`.
- `.codetalk-backups/<timestamp>/` – backup files created during `exec`.
- `stdout`/`stderr` – progress, streaming LLM output, error messages.
- VS Code webview – state and event messages.

### Caching
- **Symbol index**: in‑memory (not persisted by default); rebuilt on each scan/index.
- **Semantic cache**: stored in `{repoRoot}/.codetalk/semantic/manifest.json`; version‑stamped; used by `runSemanticMap` to avoid re‑analysis of unchanged functions. Fingerprint computed from source excerpt and context.
- **LLM response cache**: not implemented (but could be added).
- **Gitignore patterns**: loaded per scan; cached in `createGitignoreMatcher()` for file walking.
