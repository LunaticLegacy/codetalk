## Architecture

`codetalk-cli` is a command-line tool (package name `codetalk-cli`, version 0.1.3) for analyzing codebases, producing living semantic maps, and executing safe, planned changes via LLM agents. It is written in TypeScript and compiled to `dist/index.js` (two binaries: `codetalk` and `codetalk-cli`).

**Main execution path**: The user invokes a CLI command (`help`, `init`, `config`, `scan`, `map`, `ask`, `plan`, `exec`, `check`, `rollback`, `version`, `semantic`). The entry point (`src/index.ts`) parses arguments and dispatches to a handler in `src/handlers.ts`. Each handler orchestrates the required logic: config loading via `src/utils.ts`, LLM API calls via `src/llm/index.ts` (LlmPortal), tool execution via `src/tools/index.ts`, symbol indexing via `src/indexer.ts`, LSP integration via `src/lsp/`, semantic analysis via `src/semantic.ts`, progress display via `src/panel.ts`, and interactive config TUI via `src/tui/config.ts`. The file walkers honor `.gitignore` so scans, indexes, and code search stay aligned with Git's ignore rules. `scan --stream` streams the merger response; `semantic` keeps an animated worker heartbeat visible while long batches are running and each worker row shows the agent, current function, remaining queue, and the next three symbols. The LLM agent interacts through a provider‑agnostic portal (`LlmPortal`) that supports OpenAI and Anthropic handlers.

**Major components**:
- **CLI dispatcher** (`src/index.ts` + `src/constants.ts`): argument parsing, help generation, command routing.
- **Handlers** (`src/handlers.ts`): implementation of all commands, plus sub‑orchestrators for scanning, planning, execution, semantic analysis, and semantic map synchronization.
- **LLM Portal** (`src/llm/index.ts`, `src/llm/openai-handler.ts`, `src/llm/anthropic-handler.ts`, `src/llm/registry.ts`, `src/llm/types.ts`): provider‑agnostic LLM client; supports non‑streaming and streaming chat completions, tool calls, token tracking, colored stream output, error types (LlmHttpError, LlmProtocolError, LlmTimeoutError, LlmTransportError).
- **Legacy API client** (`src/api.ts`): still present but superseded by `LlmPortal`; provides `callChatCompletion`, `streamChatCompletion`, `callWithTools` wrappers.
- **Tool system** (`src/tools/`): read, grep, ls, glob, stat, git_log, search – invoked by the agent during planning/execution.
- **AST extractors** (`src/ast/`): extract symbols from TypeScript, Python, C++, and assembly files.
- **Indexer** (`src/indexer.ts`): builds in‑memory symbol indices from source trees, used by `search` and `grep` tools. Now also uses LSP pool.
- **LSP client** (`src/lsp/`): JSON‑RPC LSP client per language server; `LspPool` coordinates extraction across files/extensions.
- **Semantic analyzer** (`src/semantic.ts`): extracts detailed function/method semantics (purpose, inputs, outputs, side effects, failure modes) for the `semantic` command.
- **Prompt templates** (`src/prompts.ts`): system prompts, tool definitions, and prompts for ask, plan, semantic extraction, reviewer agents, etc.
- **Config** (`src/utils.ts`): read/write JSON config file (`~/.codetalker/config.json`), manage API URL, key, model; also `src/tui/config.ts` for interactive TUI configuration.
- **Semantic map generator** (`src/utils.ts`): produces `CODEMAP.md` from repository structure (`buildMap`, `buildTemplate`, `buildChangeSync`, `buildRepositoryEvidence`).

**Scale**: 44 source files (~280 KB total). Primary language: TypeScript (38 files). Designed for single-user CLI use, no server.

**Design philosophy**: Agent‑centric; the semantic map (`CODEMAP.md`) is the behavioral contract for LLM agents. Change safety via explicit protocol (plan → execute → sync). Rollback support via backups. Semantic analysis augments the map with function‑level metadata.

## API Surface

### CLI Commands

| Command | Purpose | Flags |
|---------|---------|-------|
| `codetalk help` | Show commands and user workflow. | (none) |
| `codetalk init` | Create a semantic map template (`CODEMAP.md`). | (none) |
| `codetalk config` | Interactive configure API URL, API key, and model. | (none) |
| `codetalk scan` | Analyze repository and produce a living semantic map. | `--json`, `--stream`, `--timeout MS`, `--depth low\|medium\|high\|full`, `--out PATH`, `--parallel N`, `--write` |
| `codetalk semantic` | Extract detailed function and method semantics into CODEMAP.md. | `--parallel N\|MAX`, `--timeout MS`, `--out PATH` |
| `codetalk map` | Generate a baseline semantic map from repository structure. | `--out PATH` |
| `codetalk ask "message"` | Answer codebase questions from map and scan context. | `--stream` |
| `codetalk plan "request"` | Generate a safe implementation plan and write to disk. | `--stream`, `--out CODEPLAN.md` |
| `codetalk exec` | Execute a CODEPLAN.md: apply all file changes in parallel via LLM (auto‑syncs map). | `--plan CODEPLAN.md`, `--parallel 4`, `--stream`, `--skip-gate`, `--diff` |
| `codetalk check` | Fail when the semantic map is missing or stale. | (none) |
| `codetalk rollback` | List backups or restore files from a backup. | `--list`, `<backup-id>` |
| `codetalk version` | Print the CLI version. | (none) |

## Execution Flows

### Startup Flow
1. User runs `codetalk <command>` (or `codetalk-cli`).
2. `dist/index.js` invokes `src/index.ts::main()`.
3. `main()` calls `utils.parseOptions(process.argv)` to extract command and flags.
4. Matches command against `constants.COMMANDS` list.
5. If no match, prints help and exits.
6. Calls the corresponding handler function from `src/handlers.ts`.
7. Handler optionally loads config via `utils.readConfig()`.
8. Handler proceeds with its specific flow.

### Command Flow
- **init**: `initMap()` → `utils.buildTemplate()` → writes to `CODEMAP.md`.
- **config**: `configure()` → `tui/config.configureTui()` → interactive menu → writes config.
- **scan**: `scanRepo()` → `utils.collectSourceFiles()` → `utils.buildScanReport()` → optionally calls `runArchitectureScan()` which uses LSP pool and LLM reviewer agents → `utils.buildMap()` → writes `CODEMAP.md`.
- **semantic**: `semanticMap()` → reads current map → calls `semantic.runSemanticMap()` which builds inventory, analyzes each function via LLM, merges cache, and writes updated map.
- **ask**: `askCodebase()` → reads map + builds evidence → sends to LLM (stream or non‑stream) → displays answer.
- **plan**: `planChange()` → reads map + builds evidence → sends to LLM with tools → writes `CODEPLAN.md`.
- **exec**: `execution()` → reads `CODEPLAN.md` → parses specs → creates backup – spawns parallel editor agents (via LLM) – validates with gatekeeper – applies changes – auto‑syncs map.
- **check**: `checkMap()` → reads map and checks if new source files are older than map; exits non‑zero if stale.
- **rollback**: `listBackups()` or `rollbackTo(id)` → restores backup directory content.
- **version**: `printVersion()` → outputs constant.

### Semantic Map Sync Flow
1. Triggered after `exec` (auto) or separately via `syncMap()`.
2. Gets list of changed files via `utils.getChangedFiles()`.
3. Sends `semanticSyncPrompt` to LLM with old map and changed files.
4. LLM returns updated map text.
5. `replaceSection()` updates `CODEMAP.md` in place.

### Error Handling
- Missing config: commands that need LLM exit with error message (or prompt for config).
- LLM API failures: caught by handler, retry logic in `callWithTools` and `callChatCompletionMessages` (limited retries).
- Tool execution failure: returns `ToolResult` with `success: false`.
- Plan execution failure: editor agents retry up to `MAX_RETRIES` (3).
- Gatekeeper rejection: loops back to coordinator for refinement, up to `MAX_GATE_RETRIES` (3).
- LSP server failure: `LspPool` catches start errors, marks server as failed, returns empty symbols for those files.
- Timeouts: all LLM calls have configurable timeout (env `CODETALKER_TIMEOUT_MS` or default 120s); tool calls may also have timeouts.

## Data Flow

### Data Sources
- **File system**: source files, config file (`~/.codetalker/config.json`), git repository.
- **LLM API**: OpenAI‑compatible or Anthropic API (configured via `apiUrl`).
- **User input**: CLI arguments, interactive prompts (config TUI, plan request text).

### Data Transformations
1. **File → Symbol Index**: `collectSourceFiles()` → `buildSymbolIndex()` (AST + LSP) → `SymbolIndex` (mapping file → exports, functions, imports, types).
2. **Symbol Index → Evidence**: `buildRepositoryEvidence()` truncates and formats for agent context.
3. **File + Map + LLM → Semantic Map**: `runSemanticMap()` → inventory items → LLM generates `SemanticFunctionRecord` per function → `renderSemanticSection()` → updated map.
4. **Plan Request → CODEPLAN.md**: `planChange()` → LLM with tools → YAML plan file.
5. **CODEPLAN.md → File Edits**: `execution()` → parse YAML → edit via LLM editor agents → apply diffs.
6. **Map + Diff → Updated Map**: `syncMap()` → LLM updates map sections.

### Data Sinks
- `CODEMAP.md` – written by `scan`, `semantic`, `map`, `syncMap`.
- `CODEPLAN.md` – written by `plan`.
- `~/.codetalker/config.json` – written by `config`, `readConfig`.
- `.codetalk-backups/<timestamp>/` – backup files created during `exec`.
- `stdout`/`stderr` – progress, streaming LLM output, error messages.

### Caching
- **Symbol index**: in‑memory (not persisted by default); rebuilt on each scan/index.
- **Semantic cache**: stored in `{repoRoot}/.codetalk/semantic/manifest.json`; version‑stamped; used by `runSemanticMap` to avoid re‑analysis of unchanged functions.
- **LLM response cache**: not implemented (but could be added).
- **Gitignore patterns**: loaded per scan; cached in `createGitignoreMatcher()` for file walking.
